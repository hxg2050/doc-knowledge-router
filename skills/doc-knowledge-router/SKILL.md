# 文档知识路由

## 职责边界

把本技能当成“图书馆管理员”。

负责：

- 判断文档应该放在哪里。
- 建立入口、目录、索引和查找路径。
- 维护每层目录的 `index.md`。
- 标注文档状态、权威范围、上下游和更新规则。
- 处理重复文档的归属和引用关系。
- 判断读者或 AI 应该先读哪份文档、再读哪份文档。
- 校验相对链接是否还能找到目标。

不负责：

- 决定文档里的业务设计结论。
- 替具体系统写设计正文。
- 规定某类文档的完整写作风格。
- 判断游戏玩法、架构方案或产品内容本身对不对。
- 把自己变成“功法设计”“战斗设计”“架构设计”等领域作者。

如果任务需要写正文内容，只负责把正文路由到正确文档；正文如何写由对应领域文档、项目规则或其他技能负责。

## 基本结构

推荐文档像图书馆一样分层：

```text
docs/
  index.md                  # 总入口：从哪里开始找
  workflows/
    index.md                # 工作入口：我要做什么
  systems/
    index.md                # 系统入口：我要看哪个系统
    <system>/
      index.md              # 单系统入口：这个系统有哪些文档
      <topic>.md            # 具体主题文档，由对应领域负责正文
  archive/
    index.md                # 归档入口：历史资料在哪里
```

原则：不要创建没有 `index.md` 的目录。

## 目录入口职责

`index.md` 是书架标签和检索卡，不是正文书本。

目录级 `index.md` 应回答：

- 这个目录收什么文档。
- 读者要找某类问题时先看哪份。
- 哪些文档是当前活跃入口。
- 哪些文档只是历史、审计或归档。
- 如果一个结论冲突，应该以哪份文档为准。

目录级 `index.md` 不应承载大量业务正文。

## 最小元信息

每个活跃入口和重要文档顶部建议有短元信息：

```md
状态：活跃 / 兼容入口 / 归档
权威范围：本文件负责哪类问题的定位或结论
上游文档：从哪里进入本文件
下游文档：本文件指向哪些文件
更新规则：什么时候应该更新本文件
```

这些元信息只用于查找和归属判断，不用于规定正文内容怎么写。

## 工作流程

当用户要求整理、查找、拆分、归档或维护文档时：

1. 先读仓库规则，例如 `AGENTS.md`、`CODEX.md` 或同类文件。
2. 找当前文档入口。优先 `docs/index.md`，没有时退回 `docs/README.md`。
3. 如果没有总入口，创建 `docs/index.md`。
4. 如果用户的问题是“我做某事该读什么”，创建或更新 `docs/workflows/index.md`。
5. 如果文档按系统或领域增长，创建或更新 `docs/systems/index.md` 和对应系统目录 `index.md`。
6. 如果一个结论在多处重复，选择唯一归属位置；其他位置只保留导航摘要和链接。
7. 如果文件已经是历史材料，把它放入或标记为归档，而不是让它继续伪装成当前权威。
8. 校验相对链接。

## 权威归属规则

当多个文档都像是在讲同一件事时，只判断“应该去哪里找”，不判断“正文内容应该怎么写”。

优先顺序：

1. 用户本轮明确指定的文档或目录。
2. 仓库级 Agent / 开发者规则。
3. `docs/index.md` 和 `docs/workflows/index.md` 的路由。
4. 与问题最匹配且标记为活跃的系统/产品文档。
5. 状态文档、审计文档、执行计划。
6. 归档文档。

如果发现冲突，应该：

- 指出冲突存在。
- 指出哪份文档应成为查找入口或权威位置。
- 建议把其他文档改成摘要、链接或归档。
- 不在本技能内重新裁决业务结论。

## 工作入口模式

`docs/workflows/index.md` 用来回答“我要做什么，该先读哪里”。

示例：

```md
## 我想改玩法状态

先读：

1. 状态变更清单
2. 架构边界文档
3. 对应系统入口

相关入口：

- 系统文档入口
- 产品内容入口
```

工作入口只写阅读路线和更新落点，不写具体玩法或架构正文。

## 系统入口模式

`docs/systems/<system>/index.md` 用来回答“这个系统有哪些文档”。

示例：

```md
# 功法系统

状态：活跃
权威范围：功法系统文档导航
上游文档：../index.md
下游文档：./skill-effects.md
更新规则：功法系统文档新增、移动、归档时更新

## 先读什么

1. [技能效果与五行收益](./skill-effects.md)

## 当前文档

| 文档 | 负责什么 |
| --- | --- |
| skill-effects.md | 技能效果与五行收益模型 |

## 相关系统

- 五行系统
- 修炼系统
```

系统入口只做导航和归属，不替系统正文做设计。

## 迁移策略

面对已经分散的 docs 目录：

1. 用 `rg --files docs` 盘点文件。
2. 用 `rg -n "^#|^##" docs` 盘点标题结构。
3. 标记每份文档属于：入口、系统导航、产品内容、状态/审计、历史归档。
4. 先建立总入口和一个小试点，不要全量搬迁。
5. 对重复内容，只确定唯一归属；旧位置保留短摘要和链接。
6. 对历史材料，标记或移动到归档区。
7. 校验链接。

迁移时尽量保持正文原意，不借整理目录之名重写业务内容。

## 链接校验

至少检查：

- 本地 Markdown 相对链接目标存在。
- 移动文件后相对链接仍正确。
- 兼容入口能指向新的总入口。
- 每个新目录都有 `index.md`。

可以用项目已有链接检查工具；没有工具时，用一次性脚本检查相对链接是否存在。

## 完成说明

完成时说明：

- 新增或修改了哪些入口。
- 哪些文档被重新归类或建立了导航。
- 哪些重复正文被改成摘要和链接。
- 是否移动或标记了归档文档。
- 链接校验结果。
- 如果只改文档，说明未运行代码测试。
